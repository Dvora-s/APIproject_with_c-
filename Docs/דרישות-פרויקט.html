סקירה כללית – איך הפרויקט עובד?

הפרויקט הוא Web API לניהול ספרייה: ספרים, מחברים, קטגוריות, חברי ספרייה והשאלות. כל בקשה מגיעה מהדפדפן (Swagger), Postman או כל לקוח HTTP, עוברת דרך שכבות, ומגיעה למסד הנתונים.

Client (Swagger/Postman) → Controller → Service → UnitOfWork → Repository → DbContext → SQL Server
Library.APIControllers, Middleware
Library.BusinessLogicServices, DTOs, Mapping
Library.DataAccessDbContext, Repositories
Library.ModelsEntities (טבלאות DB)

כל שכבה מדברת רק עם השכבה שמתחתיה – זה מבטיח קוד מסודר, ניתן לבדיקה, וקל לתחזוקה.

דרישה 1 – נושא הפרויקט (Web API מקורי)

מה נדרש? לפתח מערכת Web API ב-ASP.NET Core על נושא מקורי, עם לוגיקה עסקית אמיתית (לא רק CRUD פשוט).
איך ממומש אצלך?

נבחרה מערכת ניהול ספרייה עם לוגיקה עסקית מלאה:

  • השאלת ספר – בודק שיש עותקים זמינים ושהחבר פעיל
  • החזרת ספר – מעדכן אוטומטית את מלאי העותקים
  • מניעת מחיקת ספר/חבר עם השאלות פעילות
  • ISBN ואימייל ייחודיים במערכת

דרישה 2 – ארכיטקטורת שכבות (N-Tier / Layered Architecture)

מה נדרש? הפרויקט חייב להיות מחולק ללפחות 3–4 שכבות נפרדות (פרויקטים נפרדים ב-Solution), כאשר כל שכבה אחראית על תחום אחר.
איך ממומש אצלך? 4 פרויקטים ב-LibrarySystem.sln:
שכבהפרויקטתפקיד
Presentation (API)Library.APIControllers, Middleware, Program.cs, Swagger
Business LogicLibrary.BusinessLogicServices, DTOs, AutoMapper, חוקים עסקיים
Data AccessLibrary.DataAccessDbContext, Repositories, Unit of Work, Migrations
Models / EntitiesLibrary.Modelsמחלקות Entity (מבנה הטבלאות)

חשוב: ה-Controller לא נוגע ישירות ב-DbContext או ב-Entities – רק ב-Services.

דרישה 4 – ישויות (Entities) וקשרים בין טבלאות

מה נדרש? לפחות 4–5 ישויות, כולל:
  • Many-to-One – הרבה ספרים → קטגוריה אחת
  • Many-to-Many – ספרים ↔ מחברים (דרך טבלת קשר)
  • One-to-Many – חבר → השאלות רבות
איך ממומש אצלך? 6 ישויות:
ישותקשרים
BookMany-to-One → Category, Many-to-Many → Author, One-to-Many → Loan
AuthorMany-to-Many → Book (דרך BookAuthor)
CategoryOne-to-Many → Books
MemberOne-to-Many → Loans
LoanMany-to-One → Book, Many-to-One → Member

// Many-to-Many: Book <-> Author דרך BookAuthor
modelBuilder.Entity<BookAuthor>().HasKey(ba => new { ba.BookId, ba.AuthorId });

// Many-to-One: Book -> Category
modelBuilder.Entity<Book>()
    .HasOne(b => b.Category).WithMany(c => c.Books)
    .HasForeignKey(b => b.CategoryId);

דרישה 6 – סוגי פרמטרים (Route, Query, Body)

מה נדרש? להדגים קבלת פרמטרים משלושה מקורות:
  • Route – מזהה בכתובת, למשל /api/Books/5
  • Query – פרמטרים אחרי ?, למשל ?genre=רומן
  • Body – JSON בגוף הבקשה ב-POST/PUT
איך ממומש אצלך? 
// Route Parameter
[HttpGet("{id:int}")]
public async Task<ActionResult<BookDto>> GetById([FromRoute] int id)

// Query Parameter
[HttpGet("search")]
public async Task<ActionResult> SearchByGenre([FromQuery] string? genre)

// Body Parameter
[HttpPost]
public async Task<ActionResult<BookDto>> Create([FromBody] CreateBookDto dto)

דרישה 7 – DTOs ו-AutoMapper

מה נדרש?
  • לא לחשוף ישירות את ה-Entities ללקוח (API)
  • להשתמש ב-DTOs (Data Transfer Objects) נפרדים לקריאה, יצירה ועדכון
  • מיפוי אוטומטי בין Entity ל-DTO באמצעות AutoMapper
איך ממומש אצלך?
  • BookDto, CreateBookDto, UpdateBookDto – לספרים
  • MemberDto, CreateMemberDto, UpdateMemberDto – לחברים
  • LoanDto, CreateLoanDto – להשאלות
  • MappingProfile – ממפה Entity ↔ DTO, כולל שדות מחושבים (שם מחבר, כותרת ספר)
Library.BusinessLogic/DTOs/ Library.BusinessLogic/Mapping/MappingProfile.cs Library.API/Program.cs → AddAutoMapper 
// MappingProfile – ממפה Book ל-BookDto עם שם קטגוריה ושמות מחברים
CreateMap<Book, BookDto>()
    .ForMember(dest => dest.CategoryName, opt => opt.MapFrom(src => src.Category.Name))
    .ForMember(dest => dest.AuthorNames, opt => opt.MapFrom(src =>
        src.BookAuthors.Select(ba => ba.Author.FirstName + " " + ba.Author.LastName)));

דרישה 8 – Repository Pattern ו-Unit of Work

מה נדרש?
  • Generic Repository – פעולות CRUD גנריות לכל ישות
  • Repository ייעודי – לשאילתות מורכבות (למשל Include)
  • Unit of Work – נקודה אחת ל-SaveChanges לכל פעולה עסקית
איך ממומש אצלך?
  • IGenericRepository<T> + GenericRepository<T>
  • IBookRepository – GetByIdWithDetailsAsync, IsbnExistsAsync
  • IMemberRepository – EmailExistsAsync, GetByIdWithLoansAsync
  • IUnitOfWork – מאגד את כל ה-Repositories + SaveChangesAsync()
Library.DataAccess/Interfaces/ Library.DataAccess/Repositories/ 
// UnitOfWork – גישה מרוכזת לכל ה-Repositories
public IBookRepository Books => _books ??= new BookRepository(_context);
public async Task<int> SaveChangesAsync() => await _context.SaveChangesAsync();

דרישה 9 – Dependency Injection (הזרקת תלויות)

מה נדרש? כל השירותים וה-Repositories נרשמים ב-DI Container ומוזרקים דרך Constructor – לא נוצרים עם new בקונטרולר.
איך ממומש אצלך? ב-Program.cs: 
builder.Services.AddScoped<IUnitOfWork, UnitOfWork>();
builder.Services.AddScoped<IBookService, BookService>();
builder.Services.AddScoped<IMemberService, MemberService>();
builder.Services.AddScoped<ILoanService, LoanService>();
// ... ועוד

ה-Controller מקבל את השירות דרך Constructor:

public BooksController(IBookService bookService, ILogger<BooksController> logger)
{
    _bookService = bookService;
}

דרישה 10 – Validation (אימות קלט)

מה נדרש? אימות נתונים לפני עיבוד – שדות חובה, אורך מקסימלי, טווח ערכים, פורמט אימייל.
איך ממומש אצלך?
  • Data Annotations על ה-DTOs: [Required], [MaxLength], [Range], [EmailAddress]
  • ב-Controller: if (!ModelState.IsValid) return BadRequest(ModelState);
  • חוקים עסקיים (ISBN כפול, אין עותקים) – ב-Service Layer
// CreateBookDto
[Required(ErrorMessage = "כותרת הספר היא שדה חובה")]
[MaxLength(200)]
public string Title { get; set; }

[Range(1, 9999, ErrorMessage = "מספר עותקים חייב להיות בין 1 ל-9999")]
public int CopiesTotal { get; set; }

דרישה 11 – טיפול שגיאות גלובלי (Middleware)

מה נדרש? (סעיף 8 במסמך הדרישות)
  • Middleware מרכזי שתופס כל חריגה לא מטופלת
  • מחזיר תשובת JSON אחידה עם קוד סטטוס מתאים
  • לא לחשוף Stack Trace ללקוח
  • המערכת לא קורסת – ממשיכה לעבוד אחרי שגיאה

דרישה 13 – פעולות אסינכרוניות (async/await)

מה נדרש? (סעיף a.7 במסמך הדרישות) – כל הגישה למסד הנתונים חייבת להיות אסינכרונית עם async/await.
איך ממומש אצלך?
  • כל מתודות ה-Repository: GetByIdAsync, AddAsync, SaveChangesAsync
  • כל מתודות ה-Service: GetAllBooksAsync, CreateLoanAsync וכו'
  • כל ה-Controllers: public async Task<ActionResult> ...
// GenericRepository
public async Task<T?> GetByIdAsync(int id)
{
    return await _dbSet.FindAsync(id);
}

דרישה 14 – לוגיקה עסקית (Business Rules)

מה נדרש? חוקים עסקיים בשכבת ה-Service – לא ב-Controller ולא ב-Repository.
איך ממומש אצלך?
חוקאיפה
ISBN חייב להיות ייחודיBookService.CreateBookAsync
אימייל חייב להיות ייחודיMemberService.CreateMemberAsync
חבר לא פעיל לא יכול להשאילLoanService.CreateLoanAsync
אין עותקים זמינים – לא ניתן להשאילLoanService.CreateLoanAsync
החזרה מעדכנת CopiesAvailable +1LoanService.ReturnBookAsync
לא ניתן למחוק ספר עם השאלות פעילותBookService.DeleteBookAsync
לא ניתן למחוק חבר עם השאלות פעילותMemberService.DeleteMemberAsync
לא ניתן להקטין עותקים מתחת למושאליםBookService.UpdateBookAsync

הפרה של חוק → זריקת BusinessRuleException → Middleware מחזיר 400.
